# Панель управления VPN-подписками

Мини-приложение на Django для тестового задания Fullstack Developer: вебхук
обработки платежей + панель просмотра подписок. Без реальной интеграции с
платёжными системами, без авторизации, только моковые данные — согласно
требованиям задания.

## Стек

- Django 6 (без DRF — используются обычные представления и `JsonResponse`, так как вебхук состоит из одного простого эндпоинта, и DRF был бы избыточен для данного объёма)
- SQLite (стандартная конфигурация Django через `manage.py`, без дополнительной настройки)
- Django Templates для панели управления (без фронтенд-фреймворков, так как нужна всего одна страница)

## Как запустить

```bash
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt

cd vpn_dashboard
python manage.py migrate
python manage.py createsuperuser   # необязательно, для доступа к /admin/
python manage.py runserver
```

После запуска:

- Панель управления: http://127.0.0.1:8000/dashboard/?email=user@test.com
- Вебхук: `POST http://127.0.0.1:8000/api/payment-webhook/`
- Админ-панель: http://127.0.0.1:8000/admin/

## Пример запроса к вебхуку

```bash
curl -X POST http://127.0.0.1:8000/api/payment-webhook/ \
  -H "Content-Type: application/json" \
  -d '{"event_id":"evt_123","email":"user@test.com","amount":1000,"status":"success"}'
```

Первый запрос → `201`, создаётся подписка со статусом `active`, а поле
`expires_at` продлевается на 30 дней вперёд.

Повторный запрос с тем же `event_id` → `200` и
`"duplicate": true`, при этом новая запись `PaymentEvent` не создаётся
(дедупликация реализована через уникальное ограничение на `event_id`
и дополнительную проверку перед вставкой записи, а не только через
`get_or_create`, поэтому логика останется безопасной даже если уникальность
поля когда-либо будет убрана).

Если передать любой статус, отличный от `"success"`, запись `PaymentEvent`
всё равно сохраняется (для аудита и логирования), но состояние подписки
не изменяется.

## Панель управления

`/dashboard/?email=...` — страница только для просмотра, отображающая email,
статус подписки (active / expired / pending), тарифный план, дату окончания
подписки и состояние переключателя автопродления.

Кнопка «Продлить подписку» является моковой: она отправляет запрос на
`/dashboard/renew/`, который напрямую продлевает `expires_at` ещё на 30 дней
в базе данных. Никакие реальные платёжные сервисы при этом не вызываются.

Если параметр `?email=` не указан, отображается первый профиль из базы
данных (для удобства ручного тестирования без реализации авторизации,
которая по условиям задания явно не требуется).

## Дополнительно реализовано (из списка «если останется время»)

1. **Фильтрация в админке** — список `Subscription` в `/admin/`
   можно фильтровать по `status`, `auto_renew` и `plan_name`;
   список `PaymentEvent` фильтруется по `status` и поддерживает поиск
   по `event_id` и email.
2. **Моковая кнопка продления на фронтенде** — см. описание выше.
3. **Логирование вебхуков** — каждый вызов вебхука (новое событие,
   дубликат и активация подписки) записывается через стандартный модуль
   `logging` в консоль с именем логгера `subscriptions.webhook`.
4. **Тесты** — 5 тестов в `subscriptions/tests.py`: успешная обработка
   вебхука, дедупликация по `event_id`, валидация отсутствующих полей (400),
   проверка того, что события со статусом отличным от `success`
   не изменяют подписки, а также тест механизма автопродления только для
   подписок с `auto_renew=True`.
5. **Проверка автопродления (моковая)** — реализована функция
   `subscriptions/services.py:run_auto_renew_check()` и management-команда
   для её ручного запуска:

   ```bash
   python manage.py run_auto_renew
   ```

   Она продлевает любую подписку с `auto_renew=True`, срок действия которой
   истёк (или у которой отсутствует `expires_at`), ещё на 30 дней.
   Подписки с `auto_renew=False` остаются без изменений даже если уже
   истекли.

   Реального списания средств не происходит — это лишь имитация логики,
   которая в реальном проекте запускалась бы после колбэка от системы
   рекуррентных платежей.

Данная логика намеренно **не** подключена через Celery или cron, поскольку
это выходит за рамки задания на 2–3 часа и не входит в требования к
инфраструктуре. Вместо этого реализована ручная команда для демонстрации
работоспособности механизма.

## Структура проекта

```
vpn_dashboard/
├── manage.py
├── requirements.txt
├── vpn_dashboard/          # настройки проекта и корневые маршруты
└── subscriptions/          # единственное приложение: модели, представления, админка, тесты
    ├── models.py            # UserProfile, Subscription, PaymentEvent
    ├── services.py          # логика обработки вебхуков и автопродления
    │                        # (вынесена из views.py для удобства тестирования)
    ├── views.py             # payment_webhook, dashboard, mock_renew
    ├── admin.py
    ├── urls.py
    ├── tests.py
    ├── management/commands/run_auto_renew.py
    └── templates/subscriptions/dashboard.html
```

## Что можно было бы улучшить

- Использовать полноценные сериализаторы DRF и OpenAPI-схему для вебхука,
  если проект будет расширяться и появятся дополнительные API-эндпоинты.
- Добавить проверку подписи вебхука (HMAC) для безопасности — не реализовано,
  так как по заданию используются только моковые данные без реального
  платёжного провайдера.
- Настроить Celery Beat для автоматического запуска проверки автопродления
  вместо ручной команды.
- Добавить пагинацию и расширенный поиск по журналу `PaymentEvent`
  в админке при росте объёма данных.
- Вынести CSS в отдельный статический файл вместо встроенного блока
  `<style>` — оставлено внутри шаблона для упрощения итоговой версии проекта.
